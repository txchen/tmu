# Use an explicitly stopped per-user daemon

TMU will keep shared playback, downloads, Playlists, persistence, and application state in one long-lived per-user TMU Daemon so closing or losing a terminal does not interrupt work and multiple TUI Clients cannot create competing owners. The daemon remains alive even with no connected clients and never performs an idle shutdown; `q` quits only the current TUI Client, while the deliberately distinct `Ctrl-Q` action shows the live playback, download, and connected-client impact and requests confirmation before explicitly shutting down the daemon. Every same-user TUI Client has equal authority to request shutdown—there is no owner client—and the daemon reports an intentional remote shutdown to the other clients before disconnecting them. The normal `tmu` launch first connects to the per-user daemon and, when none exists, uses a single-winner startup lock to create it on demand before connecting; users need neither a separate daemon-start command nor launchd/systemd setup, and the next `tmu` launch starts a fresh daemon after an explicit shutdown. This favors predictable lifetime semantics and recoverability from a lost original terminal over automatically reclaiming idle resources, introducing client ownership, or exposing process management as a normal workflow.

Within a TUI Client, `q`, `Ctrl-C`, terminal hangup, and connection closure all mean Quit Client even during text entry, a modal, playback, or downloads, and therefore require no daemon-level confirmation. `Ctrl-Q` is the sole TUI shortcut for the challenged Shutdown Daemon operation. Operating-system termination delivered directly to the daemon remains a separate signal-handling path that attempts graceful persistence and cleanup.

The version-compatible `tmu daemon stop` escape hatch prints the same live impact and requires confirmation on an interactive terminal; non-interactive use is refused unless `--force` explicitly skips confirmation. Force still asks the daemon to save and shut down gracefully—it is not a PID kill—and failure of the shutdown handshake reports verified diagnostic information and manual guidance rather than automatically signaling an uncertain process.

The only other public daemon subcommand in the first version is read-only `tmu daemon status`. It never starts a missing daemon and reports daemon and protocol versions, PID and uptime, runtime path, connected-client count, shared playback identity and status, active and pending download counts, and recent recovery or persistence failure; absence returns an explicit nonzero result. Playback and editing controls remain in TUI Clients rather than growing a second command-line control interface.

The detached daemon never depends on the terminal that launched it for standard output or error. It writes bounded logs under `$XDG_STATE_HOME/tmu`, falling back to `~/.local/state/tmu`, retaining the current file and one size-rotated predecessor; status reports the path and latest severe-error summary. Logs omit full YouTube URLs, client search or input text, and full local media paths, and exclude routine playback-position ticks, concentrating on lifecycle, failed commands, download outcomes, recovery, and persistence failures.

A newly created daemon declares readiness only after configuration load, Daemon Recovery, durable snapshot restore, initial YouTube Cache scan, and socket listening succeed. Concurrent launchers wait for that single startup result, and the first client shows a startup surface until ready; timeout or fatal startup leaves a diagnostic surface with the log path and does not trigger a restart loop. External-tool health checks run after readiness, mpv remains start-on-demand, and Background Sounds remains lazy so optional capabilities do not delay a usable daemon.

TMU Config is loaded once by the daemon at startup; clients do not read independent copies, and the first version neither watches the file nor exposes hot reload. Configuration edits take effect after explicit shutdown and a subsequent `tmu` launch, while status reports the loaded path and source. Runtime identity and socket location are not user-selectable configuration, preventing accidental parallel daemons for one user.

Once confirmed, shutdown is irreversible: the daemon stops accepting Shared Commands, broadcasts a shutdown notice, cancels download preparation and active or pending batches, stops mpv, drains persistence, and exits within an approximately five-second total grace budget. After the grace period it terminates only verified owned child processes and gives the final snapshot write a bounded last attempt; failure is logged and reported nonzero rather than leaving clients waiting indefinitely. TUI Clients display a shutting-down state and then exit even if final cleanup reports failure because the daemon has already entered its terminal lifecycle state.
