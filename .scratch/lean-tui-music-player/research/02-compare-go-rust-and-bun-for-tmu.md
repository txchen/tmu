# Compare Go, Rust, and Bun/TypeScript for TMU

Research date: 2026-07-07

Status: superseded by the live decision in [Compare Go Rust And Bun For TMU](../issues/02-compare-go-rust-and-bun-for-tmu.md). The user chose Bun/TypeScript because the MVP should use subprocess-centered playback and optimize for fast iteration.

## Recommendation

Build the first TMU implementation in **Go**.

Go is the best first implementation fit because TMU's first-order risks are not "can the language render text" or "can it call HTTP APIs"; they are subprocess audio plumbing, bounded terminal redraws, provider-state boundaries, and a small distributable binary. Go has first-party process control in `os/exec`, a built-in test runner, ordinary cross-compilation knobs, and a proven local predecessor in `cliamp` that already combines Bubble Tea, Beep, tag parsing, provider clients, `yt-dlp`, and `ffmpeg` pipes ([Go `os/exec`](https://pkg.go.dev/os/exec), [Go `testing`](https://pkg.go.dev/testing), [Go install-from-source environment variables including `GOOS` and `GOARCH`](https://go.dev/doc/install/source#environment), [`cliamp` dependencies](/home/txchen/code/github/cliamp/go.mod:5), [`cliamp` `yt-dlp | ffmpeg` pipe](/home/txchen/code/github/cliamp/player/ytdl.go:272), [`cliamp` low-power ticks](/home/txchen/code/github/cliamp/ui/tick.go:5)).

Rust is the strongest long-term systems choice if TMU decides to optimize hardest for native playback, deterministic memory behavior, and maximal decode/tag ecosystem quality; Rust has Ratatui/Crossterm for TUIs, Rodio/CPAL/Symphonia for audio, Lofty for metadata, and ownership-based memory management without a runtime garbage collector ([Ratatui docs](https://docs.rs/ratatui/latest/ratatui/), [Crossterm event docs](https://docs.rs/crossterm/latest/crossterm/event/), [Rodio docs](https://docs.rs/rodio/latest/rodio/), [CPAL docs](https://docs.rs/cpal/latest/cpal/), [Symphonia docs](https://docs.rs/symphonia/latest/symphonia/), [Lofty docs](https://docs.rs/lofty/latest/lofty/), [Rust ownership chapter](https://doc.rust-lang.org/book/ch04-01-what-is-ownership.html)). The cost is implementation complexity: TMU would be solving queue state, audio thread state, process lifecycle, async provider clients, and TUI event flow while also paying Rust's ownership and ecosystem-integration tax.

Bun/TypeScript is the wrong first implementation for TMU's core player. Bun has good TypeScript ergonomics, a built-in test runner, subprocess APIs, and standalone executable support, but its first-party runtime surface is not a native audio playback surface; the realistic playback path is external players/processes or native add-ons, which weakens TMU's "lean native terminal player" goal ([Bun TypeScript docs](https://bun.com/docs/runtime/typescript), [Bun test docs](https://bun.com/docs/test), [Bun child process docs](https://bun.com/docs/runtime/child-process), [Bun standalone executable docs](https://bun.com/docs/bundler/executables), [Bun Node-API docs](https://bun.com/docs/runtime/node-api)).

## Recommended First-Implementation Shape

Use Go with a deliberately smaller scope than `cliamp`: Bubble Tea for the event loop, Beep/Oto for the speaker/audio abstraction, `ffmpeg` as a fallback decoder for formats that are awkward in Go, `yt-dlp` only for download/cache workflows rather than live YouTube playback, and provider clients as plain interfaces around HTTP/JSON APIs ([Bubble Tea docs](https://pkg.go.dev/charm.land/bubbletea/v2), [Beep docs](https://pkg.go.dev/github.com/gopxl/beep/v2), [Oto docs](https://pkg.go.dev/github.com/ebitengine/oto/v3), [`cliamp` ffmpeg fallback](/home/txchen/code/github/cliamp/player/decode.go:249), [`cliamp` `yt-dlp | ffmpeg` live pipe](/home/txchen/code/github/cliamp/player/ytdl.go:272), [OpenSubsonic `stream` endpoint docs](https://opensubsonic.netlify.app/docs/endpoints/stream/)).

The low-power rule should be an architecture rule, not a theme option: no always-on visualizer in the MVP, no fixed 30/60 FPS redraw loop, and no playback progress updates faster than needed for user-visible state; `cliamp`'s `TickLowPowerPlaying` and `TickIdle` show the exact kind of cadence TMU should preserve while deleting the high-frequency visualizer modes from the MVP ([`cliamp` tick constants](/home/txchen/code/github/cliamp/ui/tick.go:5)).

Keep Rust as the likely second-pass rewrite or audio-core extraction candidate if Go's audio/tag stack becomes the limiting factor, because Rust's Rodio/CPAL/Symphonia/Lofty set is materially stronger for native media work than Go's common stack ([Rodio docs](https://docs.rs/rodio/latest/rodio/), [CPAL docs](https://docs.rs/cpal/latest/cpal/), [Symphonia docs](https://docs.rs/symphonia/latest/symphonia/), [Lofty docs](https://docs.rs/lofty/latest/lofty/)).

Use Bun/TypeScript for optional provider experiments or tooling only, not for the player core, because TMU's differentiator is a lean native terminal player and Bun's best-documented strengths are TypeScript execution, subprocesses, tests, and standalone packaging rather than native audio output ([Bun TypeScript docs](https://bun.com/docs/runtime/typescript), [Bun child process docs](https://bun.com/docs/runtime/child-process), [Bun test docs](https://bun.com/docs/test), [Bun executable docs](https://bun.com/docs/bundler/executables)).

## Main Risks With The Go Choice

1. **Audio format coverage may require `ffmpeg` anyway.** `cliamp` already treats AAC, ALAC, WMA, Opus, WebM, and related extensions as requiring `ffmpeg`, so TMU should be explicit that single-binary distribution does not automatically mean "no runtime helper binaries" unless the MVP limits local formats more aggressively ([`cliamp` supported extensions](/home/txchen/code/github/cliamp/player/decode.go:25), [`cliamp` `needsFFmpeg`](/home/txchen/code/github/cliamp/player/decode.go:227), [FFmpeg documentation](https://ffmpeg.org/ffmpeg.html)).

2. **Process lifecycle is a real domain boundary.** `cliamp` has dedicated code for `yt-dlp` availability, install hints, `ffmpeg` availability, subprocess stderr capture, timeout handling, and child cleanup, so TMU should isolate playback subprocesses behind a tested port instead of spreading `exec.Command` calls through UI/provider code ([`cliamp` install hints](/home/txchen/code/github/cliamp/player/ytdl.go:76), [`cliamp` monitorExit](/home/txchen/code/github/cliamp/player/ytdl.go:252), [`cliamp` cleanup](/home/txchen/code/github/cliamp/player/ffmpeg.go:232), [Go `os/exec`](https://pkg.go.dev/os/exec)).

3. **Provider logic can sprawl quickly.** `cliamp`'s Go tree has provider packages for many services and a Navidrome client with browsing, search, caches, timeouts, and API error handling; TMU should start queue-first with local, Navidrome/OpenSubsonic, and offline YouTube cache only ([`cliamp` Navidrome client](/home/txchen/code/github/cliamp/external/navidrome/client.go:25), [`cliamp` YouTube Music provider base](/home/txchen/code/github/cliamp/external/ytmusic/provider.go:1), [OpenSubsonic `getAlbumList2` docs](https://opensubsonic.netlify.app/docs/endpoints/getalbumlist2/)).

4. **Go's GC is acceptable only if allocation and redraw budgets are enforced.** Go is garbage collected, and `cliamp`'s visualizer benchmark and low-power tick code show that terminal animation/analysis can become a measurable workload, so TMU should test allocations and idle CPU early ([Go GC guide](https://go.dev/doc/gc-guide), [`cliamp` visualizer benchmark](/home/txchen/code/github/cliamp/ui/visualizer_bench_test.go:31), [`cliamp` low-power ticks](/home/txchen/code/github/cliamp/ui/tick.go:12)).

## Bottom Line

Choose **Go** for TMU v0 because it gives the shortest path to a lean, testable, event-driven, distributable terminal player while preserving the most useful lessons from `cliamp`. Choose **Rust** only if the project is willing to trade first-version speed for a stronger media-native core. Do **not** choose **Bun/TypeScript** for the first player core; it is better reserved for provider prototypes or developer tooling.
